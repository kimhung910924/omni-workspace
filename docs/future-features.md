# 향후 기능 메모 (지금 구현 안 함, 잊지 않으려고 기록만)

## 워크스페이스 개수 제한 (Pro 모드)
- 무료: 워크스페이스 1~2개까지 생성 가능
- Pro 모드: 워크스페이스 최대 10개까지 생성 가능
- 워크스페이스가 하나도 없을 때: 기본 첫 화면(웰컴/온보딩) 표시
- 구현 전제조건: omni-backend(Supabase)에서 구독 상태(entitlement) 판단 로직 필요
- 착수 시점: 단일 워크스페이스(탭+URL 저장/복원) 동작이 완전히 검증된 뒤

## 상단탭 전환 시 즉시 전환 (webview 재로드 없이) — ProMax 후보

**문제 제기 배경**: 지금 구조(탭은 트리거, Stage/Dock은 전역 싱글톤)는 탭을
전환하면 해당 탭의 webview들이 다시 로드되어 시간이 걸림. 무료/Pro(4탭 x
8슬롯=32개 한도)에서 전체를 항상 메모리에 띄워두면 즉시 전환이 될 텐데,
이걸 ProMax 차별 기능으로 만들 수 있는지 논의함.

**결론: "32개 전부 상시 유지"는 기본값으로 채택하지 않음.** 대신 "스마트
유지 / Hot Workspace Cache" 형태로 ProMax에 넣는 것을 후보로 남김.

### 왜 32개 전부 상시 유지가 위험한가
- Claude/ChatGPT/Gemini/Grok/Perplexity 전부 무거운 SPA, webview 1개가
  실질적으로 브라우저 탭 1개와 비슷한 비용. 슬롯당 대략 200~600MB+ 추정,
  32개면 수GB~20GB대까지 갈 수 있음 (추정치, 실측 필요)
- `display:none`은 화면에 안 그릴 뿐 JS 타이머/세션/네트워크/웹소켓이 계속
  살아있어 메모리뿐 아니라 CPU도 누적 소모
- 저사양 PC에서 먹통/발열/배터리드레인 리스크, 고사양에서도 "묵직해짐"
  체감 가능

### 더 결정적인 이유 — 이미 한 번 실패한 접근법과 같은 방향
- `docs/IN-PROGRESS.md`의 "[폐기된 접근법] Stage 3 — 탭 전환 시 webview
  유지(retention)" 항목과 본질적으로 동일한 방향. 그때 비활성 탭 URL 갱신
  안 됨 → partition 충돌 → Tab B 슬롯이 Tab A에도 나타나는 심각한 회귀까지
  나서 전체 revert함
- 그 실패의 핵심 원인은 "provider당 파티션을 공유하는 슬롯 여러 개가 동시에
  살아있을 때 navigation 이벤트가 자기 소속 탭을 헷갈림" — 이 구조적 문제를
  먼저 풀지 않으면 탭 레벨 retention은 다시 같은 버그를 낼 가능성 높음
- 그래서 단순 "display:none으로 숨기기만 하면 됨"이 아니라, **webview
  생명주기 관리자**를 별도로 설계해야 함 (탭 단위가 아니라 슬롯 단위 전역
  풀 + LRU eviction 방향)

### 추천 설계 방향 (구현 시 참고용, 아직 미설계)
- 이름 후보(내부용): Hot Workspace Cache / Fast Switch Mode
- "무제한 유지"가 아니라 "최근 사용 우선순위 기반 캐시":
  - 항상 유지: 현재 활성 탭 + 그 안의 슬롯
  - 추가 유지: 직전 탭 1개, 또는 사용자가 고정(pin)한 워크스페이스
  - 나머지: URL/슬롯구성/Dock·Stage 상태만 저장, 실제 webview는 unload →
    재진입 시 재로드
- 설정에 "성능 모드" 노출 고려: 메모리 절약 / 균형(기본) / 빠른 전환 /
  최대 유지(실험적, 고사양 전용)
- 살아있는 webview 수 상한 권장치(추정, 실측 후 조정 필요):
  무료 2~4 / Pro 8 / ProMax 기본 12~16 / ProMax 고성능 옵션 24, 32 전체는
  숨겨진 실험 옵션 정도로
- 가칭 타입: `RetentionPolicy { maxLiveWorkspaces, maxLiveSlots,
  keepPreviousWorkspace, pinnedWorkspaceIds, unloadAfterMinutes }`
- 마케팅 문구 주의: "32개 항상 켜둠"이 아니라 "최근 워크스페이스를 메모리에
  유지해 전환을 빠르게 함" 식으로 표현 — 약속한 숫자를 못 지키면 역효과

**구현 전제조건**: 실제 RAM 점유량 실측(현재 8슬롯/2탭 기준) 먼저 해보고
숫자 감을 잡은 뒤 설계 착수. 폴더 구조 정리, 웹슬롯 3차 시도보다 우선순위
낮음 — 별도로 시간 잡고 "측정 → 설계 → 작은 단위 검증" 순서로 진행할 것
- 착수 시점: Pro 티어(Supabase entitlement) 구현 이후, ProMax 티어 설계
  시점
