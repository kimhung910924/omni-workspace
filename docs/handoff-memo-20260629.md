# 인수인계 메모 (2026-06-29 작성)

## 현재 코드 상태

- `main` 브랜치, 깨끗한 상태
- 웹슬롯 관련 코드는 main에 없음 (2차 시도까지 전부 폐기됨, 커밋된 적 없음)
- main에는 i18n, Slot/Stage/Dock, 워크스테이션 6단계, OAuth 수정 등만 들어있음

## 중요 — 2차 시도 "실패"의 진짜 원인이 코드가 아니었음이 밝혀짐

자세한 기록은 `docs/IN-PROGRESS.md`의 "[해결됨, 2026-06-29] 진짜 원인: dev
서버 포트 충돌" 항목 참고. 핵심만 요약:

- 2차 시도 때 "워크스테이션 소실 + Claude 로그아웃" 증상이 재발해서 브랜치를
  통째로 폐기했었음
- 그 뒤로 코드를 전부 되돌렸는데도(맥북 src.zip과 한의원 src.zip을 직접 diff
  비교해서 100% 동일함까지 확인) 맥북에서 똑같은 증상이 계속 재발해서 추적을
  이어갔음
- **진짜 원인: `package.json`의 dev 스크립트가 `VITE_DEV_SERVER_URL=
  http://127.0.0.1:5173`을 하드코딩하고 있는데, 이전 `npm run dev`가 깔끔하게
  종료되지 않아 5173 포트를 좀비 프로세스가 점유 → Vite가 5174로 밀림 →
  Electron은 그래도 5173을 로드 시도 → 렌더러가 제대로 안 뜸 → 워크스테이션
  /Claude 전부 깨진 것처럼 보임**
- `lsof -ti:5173 | xargs kill -9` 후 재시작하면 정상 복구됨 — 맥북에서 직접
  확인 완료
- **즉 2차 시도에서 작성한 웹슬롯 코드 자체는 결함이 없었을 가능성이 높음.**
  당시 의심했던 `workspaceStore.ts`, `groupStore.ts`는 둘 다 무죄로 결론

## 다음 할 일 — 웹슬롯 3차 시도

- 2차 시도 때 만든 구조(`AiSlot`/`WebSlot` discriminated union,
  `WEB_SLOT_PARTITION`, `isAiSlot()` 분기, `isMainFrame` 가드)를 그대로
  재구현하면 됨 — 처음부터 새로 설계할 필요 없음, 새로 디버깅할 필요도 없음
- 단, 코드는 커밋 안 하고 폐기했으므로 실제로는 다시 타이핑/구현해야 함
  (Codex 프롬프트는 기존 1단계 프롬프트 + 핫픽스 프롬프트 내용을 합쳐서
  재사용 가능)
- **작업 시작 전 그리고 매번 재시작해서 테스트할 때마다, 코드를 의심하기
  전에 먼저 터미널에 `Port 5173 is in use, trying another one...`이
  뜨는지부터 확인할 것.** 뜬다면 `lsof -ti:5173 | xargs kill -9` 후 재시작
  해서 증상이 사라지는지 먼저 보고, 그래도 남는 증상만 진짜 코드 문제로
  취급할 것
