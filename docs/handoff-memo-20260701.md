## 이번 세션 요약 — Gemini 워크스테이션 재오픈 복원 실패, 최종 결론

**결론: 우리 코드 문제가 아니라 Google Gemini 자체 버그. 우리 쪽 작업 종료.**

자세한 진단 과정은 `docs/CHANGELOG-DEV.md`의 "2026-07-01 — Gemini 워크스테이션
재오픈 복원 실패, 원인을 Google 자체 버그로 최종 확정" 항목 참고.

핵심 증거: 실패 상태의 Gemini Gem 딥링크 URL을 그대로 복사해서
1) 우리 앱(Electron webview), 2) 순정 Chrome 주소창, 3) Chrome 자체 탭그룹
— 세 환경 모두에서 동일하게 "삭제된 잼으로 생성된 대화입니다" → 채팅 목록으로
튕기는 현상이 재현됨. Chrome 자체 탭그룹(Google 1st-party 브라우저의
1st-party 기능)까지 동일하게 실패한다는 게 결정적 — 이건 Electron이나
webview나 UA나 우리 코드 구조와 무관하다는 뜻.

이 과정에서 시도했다가 실측으로 반증되거나 폐기된 것:
- UA(User-Agent) 스푸핑 — 제미나이가 스스로 제안한 가설이었으나, "F5로는
  거의 안 살아나고 앱 재시작으로는 가끔 살아난다"는 관찰과 애초에 모순
  (UA는 재시작해도 동일). 크롬 탭그룹 재현으로 최종 기각.
- selector 기반 "대화 DOM 렌더링 여부" 판정 — 라이브 DevTools로 직접
  검증했더니 정상 대화에서도 `false`만 나옴. Angular 커스텀 태그 이름을
  추측한 게 틀렸음.
- `document.body.innerText.length` 기반 임계값(2,500자) + reload →
  loadURL(no-cache) → reloadIgnoringCache 3단계 에스컬레이션 재시도 —
  판정 로직 자체는 실측 데이터(성공 10,924자 vs 실패 853/1,160자)로
  검증까지 끝냈으나, 원인이 Google 버그로 확정되면서 **적용하지 않기로
  결정**. 지시문 파일은 만들어졌으나 Codex에 넘기지 않고 폐기.

**메모리(memory_user_edits)에도 이 결론을 기록해둠** — 다음 세션에서 같은
가설을 또 시도하는 일이 없도록. "제미나이 젬 복원 왜 안 됨" 질문이 다시
나오면 이 메모리를 참고해서 바로 "우리 문제 아님, 확정됨"으로 답할 것.
